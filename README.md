# Contact Center Agent

A FastAPI service that handles customer support conversations. Each chat turn classifies the customer's intent, checks whether the conversation should be escalated to a human agent, generates a response via a local LLM, and records metrics against the conversation state.

## Stack

- **FastAPI** — HTTP server and request validation
- **Pydantic** — request/response schemas
- **pytest** — test runner
- **Ollama** — local LLM runtime; the responder sends requests to Ollama's OpenAI-compatible endpoint (`http://localhost:11434/v1`) using `llama3.2`

## Prerequisites

Ollama must be running locally with `llama3.2` pulled before starting the server:

```bash
brew install ollama
brew services start ollama
ollama pull llama3.2
```

## Setup

```bash
python3 -m venv venv
source venv/bin/activate
pip install -r requirements.txt
```

## Running the server

```bash
source venv/bin/activate
uvicorn app.main:app --reload
```

## API

### `GET /health`

```bash
curl http://localhost:8000/health
```

```json
{"status": "ok"}
```

---

### `POST /chat`

Accepts a message from a customer, classifies the intent, checks escalation rules, and returns a response generated by `llama3.2`.

```bash
curl -X POST http://localhost:8000/chat \
  -H "Content-Type: application/json" \
  -d '{
    "conversation_id": "conv-001",
    "message": {"role": "customer", "text": "I was charged twice this month"}
  }'
```

```json
{
  "conversation_id": "conv-001",
  "response_text": "I understand your concern about the duplicate charge...",
  "escalated": false
}
```

`escalated` is `true` when any of these conditions are met:
- Classifier confidence < 0.4
- Conversation sentiment score < -0.5
- Failed attempts >= 3

---

### `GET /metrics`

Returns aggregate stats across all conversations in the current server process.

```bash
curl http://localhost:8000/metrics
```

```json
{
  "total_conversations": 4,
  "average_messages_per_conversation": 2.5,
  "escalation_rate": 0.75,
  "average_sentiment_score": -0.2,
  "intent_distribution": {
    "Billing": 5,
    "Technical": 2,
    "General Question": 1
  }
}
```

## Architecture

Each `POST /chat` request flows through five modules in sequence:

```
main.py
  └── state_manager.py   get or create ConversationState for the conversation_id
  └── classifier.py      predict intent and confidence from the message text
  └── escalation.py      apply three deterministic rules → (bool, reason)
  └── responder.py       build messages with intent-specific system prompt, call llama3.2
  └── state_manager.py   write confidence, escalated, intent_history back onto state
```

`ConversationState` (in `state_manager.py`) is the shared data structure that accumulates across turns:

| Field | Type | Purpose |
|---|---|---|
| `messages` | `list[str]` | Raw message texts in order |
| `intent_history` | `list[str]` | Intents classified per turn |
| `confidence` | `float` | Classifier confidence from the last turn |
| `sentiment_score` | `float` | Set externally; drives escalation |
| `failed_attempts` | `int` | Drives escalation |
| `escalated` | `bool` | Whether the last turn triggered escalation |
| `last_intent` | `str \| None` | Most recent intent (unused by current logic) |

`metrics.py` reads directly from `conversation_store` (the in-memory dict) and computes all values on demand.

## Tests

```bash
# Run all tests
python -m pytest

# Run a single file
python -m pytest tests/test_escalation.py -v

# Run a single test
python -m pytest tests/test_escalation.py::test_low_confidence_triggers_escalation -v
```

### TDD approach

All modules were implemented test-first. There are 43 tests across 6 files:

| File | Tests | What it covers |
|---|---|---|
| `test_api.py` | 3 | HTTP status codes and response shapes for all three endpoints |
| `test_classifier.py` | 1 | Classifier interface contract |
| `test_escalation.py` | 13 | All three escalation conditions, boundary values, reason strings |
| `test_metrics.py` | 12 | Each computed metric in isolation, empty store, multi-conversation |
| `test_responder.py` | 9 | Intent-to-system-prompt routing, history passing, API error handling |
| `test_main_wiring.py` | 5 | State write-back after each turn (confidence, escalated, intent_history) |

All collaborators (classifier, escalation, responder) are mocked in `test_main_wiring.py`. The Ollama client is mocked in `test_responder.py`. No tests make real network calls.

## Limitations

- **Classifier is a stub** — `IntentClassifier.predict()` always returns `intent="General Question"` with `confidence=0.0`. Because confidence is always below the 0.4 threshold, every conversation currently escalates. A real implementation would replace this with keyword matching or an ML model.
- **Sentiment score is never updated** — `ConversationState.sentiment_score` defaults to `0.0` and nothing in the current request pipeline writes to it. The sentiment escalation rule is implemented but never triggers.
- **In-memory state only** — `conversation_store` is a plain Python dict on the server process. All state is lost on restart. There is no database or cache backing.
- **No authentication** — all endpoints are unauthenticated.
- **Evaluation script is a stub** — `app/evaluation.py` prints a message and exits.
